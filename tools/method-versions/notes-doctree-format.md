# doctree `_builtin/*.md` format notes

Generated by `tools/parse_doctree.rb` against `doctree/manual/api/_builtin/*.md`
(148 files, 2026-07-23 checkout). Output: `doctree-entries.tsv`, 3160 rows (one row
per `### def` / `### module_function def` / `### const` / `### gvar` signature
line).

## 1. Heading forms found (via `grep -hn "^#" *.md` then classified)

`### ` headings split into two disjoint groups:

- **Signature headings** (matched by `mdparser.rb` `SIG_RE`, bitclust
  `lib/bitclust/mdparser.rb:51`): `### def NAME...`, `### module_function def
  NAME...`, `### const NAME...`, `### gvar $NAME...`. Counts: `def`=2520,
  `module_function def`=249, `const`=335, `gvar`=56 → **3160 total**, matching
  the tool's row count exactly (independent grep cross-check, see §6).
- **Prose H3 subsections** (103 lines): plain titles like `### シンボルの実装と用途`
  or `### 手続きを中断して値を返す {#anchor}` inside `Symbol.md`, `lambda_proc.md`,
  `printf-format.md`, `thread.inc.md`, `constants.md` (`### 注意`), etc. These
  never match `SIG_RE` and produce no TSV rows.

`# ` (H1) headings: only lines of the literal form `# class X < Y` / `# module
X` / `# object X` / `# object X < Y` / `# reopen X` / `# redefine X` are real
entity headers (`mdparser.rb` `parse_level1_header`, rrdparser.rb:196-202-ish
mirrored in `mdparser.rb`). Many other lines start with a single `#` but are
Ruby comments *inside* fenced code samples (e.g. `ARGF.md:45`, `Module.md:880`
`# class C` inside a ` ```ruby ` block) — the tool tracks fenced-code state
(mirroring `mdparser.rb#md_break`, lines 72-89) so these are correctly not
treated as headers.

`## ` (H2) headings found, all matching one of `mdparser.rb`'s
`read_level2_blocks` (lines 263-285) patterns: `Class Methods`, `Instance
Methods`, `Public/Private/Protected Instance Methods`, `Singleton Methods`,
`Private Singleton Methods`, `Methods` (bare), `Module Functions`,
`Constants`, `Special Variables`. One instance has a trailing space
(`## Singleton Methods `) — handled via `.strip`.

### Multi-entity files

Some files declare more than one class/module via H1 headers sequentially;
`class_name` is re-derived every time a new one is seen. Most extreme case:
`SystemCallError.md` has **152** H1 headers (`SystemCallError`, `module
Errno`, then ~150 `Errno::EXXX`/`Errno::EPERM`/... subclasses), but only the
first two contexts (`SystemCallError`, `Errno::EXXX`) actually contain
`### def|const` headings — the ~150 per-errno subclasses are pure prose.
Verified: all 8 real signature lines in that file map to `class_name` in
`{SystemCallError: 5, Errno::EXXX: 3}` (checked by cross-tab against the tool
output). Other multi-entity files: `Encoding.md` (6), `Fiber.md` (2:
`FiberError`, `Fiber`), `Math.md` (2: `Math`, `Math::DomainError`), `Module.md`
(2: `Module`, plus `# class C` — but that one is inside a fence, ignored).

### `# object` entities

`ARGF.md`, `ENV.md`, `main.md` use `# object NAME` with **no `## ...` H2 at
all**. Per `mdparser.rb#read_object_body` (lines 252-261), the context
defaults to `type = :singleton_method` immediately, before any H2. All 85
signature lines across these 3 files come out as `kind=s`, verified.

## 2. `{: since=}` / `{: until=}` attribute lines

Regex: `/\A\{:(.*)\}[ \t]*\z/` (`rrdparser.rb:274` `METHOD_ATTRIBUTE_LINE_RE`).
Binds **only to the immediately preceding signature line** (no blank line in
between) — `mdparser.rb#read_chunk` (lines 297-320) stops collecting header
lines at the first line that is neither `SIG_RE` nor an attribute line; the
name resolution / per-signature binding is `rrdparser.rb#method_attributes`
(lines 545-595), keyed by `method_attribute_target_name` (line 624-629) which
re-parses each signature line as it's seen (so 3 consecutive `### def any?
...` + `{: since=""}` pairs each bind independently — confirmed with
`Array.md`'s `any?`, see §5).

Corpus totals: **109** attribute lines total.
- Tokens: `since="X"` (95: 75 `EMPTY` i.e. `since=""`, 13× `1.9.1`, 7×
  `1.9.3`), plus flag-only tokens `{: undef}` (6, all in `Complex.md`:
  lines 132/135/138/141/144/147) and `{: nomethod}` (8, all in `Object.md`:
  lines 323/370/398/426/452/480/505/522). 95 + 14 = 109 — exact match.
- **No `until="X"` attribute exists anywhere in the corpus** (0 occurrences;
  `until_attr` is `-` for all 3160 rows). `until=""`/other `until=` values may
  appear in future docs; the parser supports them identically to `since=`.
- No line combines `since=` and `until=` together, and no signature has
  conflicting attribute values across its own multiple occurrences (checked;
  not formally asserted by the tool but consistent throughout).
- **Design note**: `since_attr`/`until_attr` in the TSV reflect the attribute
  line immediately following *that physical heading occurrence* only — not
  bitclust's runtime aggregation-by-name (bitclust ties `since_by_name` to the
  *name*, so if one of several overloads sharing a name has an explicit
  attribute and a sibling overload doesn't, bitclust's `MethodEntry` would
  still expose that value for the name as a whole). This didn't matter for
  any case in this corpus — every same-named signature group that has any
  attribute has one on *every* occurrence (e.g. `Array#any?`'s 3 overloads
  each carry their own `{: since=""}`).

## 3. Preprocessor directives (`#@since`/`#@until`/`#@if`/`#@else`/`#@end`/`#@include`/`#@#`)

Verified in `bitclust/lib/bitclust/preprocessor.rb` (lines 82-139): the
preprocessor is a **blind line-based filter** — it does not know about
Markdown code fences, so `#@` directives are recognized and processed
identically whether they sit in prose or inside a ` ```ruby ` block (e.g.
`Array.md:1929-1943` has `#@since 3.4`/`#@else`/`#@end` *inside* a fenced
sample, toggling only the expected-output text, not the fence itself). The
tool's condition-stack tracking (push on `#@since`/`#@until`/`#@if`, invert on
`#@else`, pop on `#@end`) is applied to every line regardless of fence state;
fence tracking is applied *separately* only to decide whether a line can be a
structural heading (H1/H2/signature/attribute).

- 55 of 148 files contain at least one `#@since`/`#@until`/`#@if` directive.
- Two usage patterns: (a) wrapping sample **output text only** (majority —
  e.g. `NoMethodError.md:10-18` 3-way `#@if` branch on Ruby's changing
  `NoMethodError` message quoting; `Array.md:1929-1943` transpose example) —
  these never enclose a `### ` heading, so they don't affect any TSV row;
  (b) wrapping an actual **heading** or **heading + body** — these produce a
  non-`-` `pp_cond`. Found 141 such rows. Examples:
  - `Array.md:1022-1047` `#@since 3.4` around `### def fetch_values` (2
    overload lines) → `pp_cond=since:3.4`.
  - `Fiber.md:123-125` / `:170-173` `#@since 3.1` around `#@include(Fiber.current)`
    / `#@include(Fiber.transfer)`+`#@include(Fiber.alive_p)` → the *fragment
    files'* rows (`Fiber.current.md:1`, `Fiber.transfer.md:1`,
    `Fiber.alive_p.md:1`) get `pp_cond=since:3.1`, inherited through the
    include boundary (see §4).
  - `String.md:3862-3868` nested `#@since 3.2` (outer, opened earlier) +
    `#@since 3.3` (inner, around one `bytesplice` overload each) →
    `pp_cond=since:3.2,since:3.3` (nesting is comma-joined outermost-first,
    per the task spec's example format).
  - `RubyVM__YJIT.md:69-76` — outer `#@since 3.3` plus a 3-way `#@if` on finer
    Ruby-version ranges around 3 different `enable` signatures →
    `since:3.3,if:("4.0" <= version)` etc.
- `#@else` semantically inverts the current condition. Since bitclust's
  `#@since X` builds `version >= "X"` and `#@until X` builds `version <
  "X"` (`preprocessor.rb:111-114`), the else-branch of a `#@since X` is
  exactly `#@until X`'s condition and vice versa — the tool represents this
  by literally swapping the label (`since:X` <-> `until:X"`) rather than
  inventing a third notation. `#@if(EXPR)`'s else becomes `if_not:EXPR` (the
  boolean is never evaluated). This means `until:X` in `pp_cond` may come from
  either a literal `#@until X` (32 raw occurrences in the corpus) or an
  inverted `#@since X`/`#@else` — the two are indistinguishable in the output
  by design, since they mean the same thing.
- `#@if` raw expression forms seen (kept verbatim after `#@if`): `version <
  "3.3"`, `"3.3" <= version and version < "3.4"`, `"3.4" <= version`, `(version
  == "3.1")`. No boolean evaluation is performed.
- `#@#...` = preprocessor comment, always stripped, no structural effect
  (`preprocessor.rb:90-91`).
- 0 warnings emitted by the tool: no unterminated `#@if`/`#@since`/`#@until`
  at EOF of any top-level file, no unclosed fence at EOF, no unknown `#@`
  directive, no unknown `## ` header, no un-extractable signature name.

### `#@include`

8 distinct fragment targets, some included repeatedly: `Struct.attention`
(×16, `Struct.md`), `Data.attention` (×9, `Data.md`), `printf-format` (×2:
`functions.md`, `String.md`), `pack-template` (×2: `Array.md`, `String.md`),
`lambda_proc` (×2: `Proc.md`, `functions.md`), plus single uses of
`BasicObject.public_methods_from_Object`, `BasicObject.private_methods_from_Object`
(`BasicObject.md`), `Fiber.current`, `Fiber.transfer`, `Fiber.alive_p`
(`Fiber.md`), `functions`, `specialvars` (`Kernel.md`), `Module.attr`,
`Module.include`, `Module.prepend`, `Module.alias_method`,
`Module.define_method`, `Module.remove_method`, `Module.undef_method`
(`Module.md`), `constants` (`Object.md`), `functions_pp` (nested inside
`functions.md`), `thread.inc` (`Thread.md`). All 22 target files were
resolved successfully (0 "include target not found" warnings); the 4
remaining front-matter-less files (`Complex__compatible.md`,
`NameError__message.md`, `Rational__compatible.md`, `Warning__buffer.md`) are
never `#@include`d anywhere and are pure `#@# nodoc` stubs with zero content
lines — the tool still processes them as (harmless, 0-row) top-level files.

The tool resolves each `#@include(NAME)` by expanding `NAME.md`'s lines
in-place, **sharing the current mutable context** (`class_name`, H2-derived
`type`, code-fence state) and the **current `pp_cond` stack** (so a fragment
inherits whatever conditions wrap its `#@include` line, then may push/pop its
own on top — this is equivalent to bitclust's actual behavior, where
`Preprocessor#next_line`'s `#@include` branch only recurses `unless
current_cond.processing?` is false, i.e. the whole include is skipped when the
outer condition is false, which has the same net effect as AND-ing the
conditions). `class_since`/`class_until` are likewise inherited from the
*top-level* file's front matter, not the fragment's own (fragments never have
front matter). This was necessary for correctness: e.g. `BasicObject.md` has
`since: "1.9.1"` in front matter and `#@include`s
`BasicObject.public_methods_from_Object.md` / `...private_methods_from_Object.md`
— those fragment rows correctly show `class_since=1.9.1`.

Files that are `#@include` targets are excluded from top-level iteration (to
avoid duplicate/mis-attributed rows); this was verified safe because every
repeatedly-included fragment with real content is included exactly once
(`Fiber.*`, `BasicObject.*`, `Module.*`, `constants.md`, `functions.md` →
`specialvars.md`, `functions_pp.md`), while the fragments included multiple
times (`Struct.attention`, `Data.attention`, `pack-template`, `printf-format`,
`lambda_proc`) contain **zero** `### def|const|gvar|module_function def`
headings (prose/anchors only), so repeated inclusion cannot double-count real
rows.

## 4. `class_name` derivation

Primary source: the nearest preceding real H1 entity header, tracked
sequentially through the file (and through `#@include` expansion, inheriting
whatever `class_name` the includer had at that point). Fallback (only used
before the first H1 in a file, never actually observed to reach a real
signature line in this corpus — 0 rows have `class_name="?"`, the internal
sentinel): filename with `.md` stripped and `__` → `::` (e.g.
`Enumerator__Yielder` → `Enumerator::Yielder`).

## 5. `kind` derivation — important finding: keyword text is *not* authoritative

Traced through `bitclust/lib/bitclust/mdparser.rb#method_signature` (lines
324-344) and `rrdparser.rb#method_id`/`Context#signature` (lines 476-479,
632-639): the `def `/`module_function def `/`const `/`gvar ` keyword prefix is
stripped and replaced with a generic `--- ` before signature parsing
(`SIG_RE`/`MD_METHOD_SIG_PREFIX_RE`). The resulting `Signature` only carries an
explicit typemark when the remaining text has its own class-path+typemark
prefix (never happens in this corpus) or — critically — when the name starts
with `$` (gvar), which forces the `GVAR` regex branch
(`Signature.new(nil, '$', name[1..])`, rrdparser.rb:343-344) regardless of the
literal keyword used. For plain identifiers (`def`, `const`, `module_function
def` all produce the *same* shape after stripping), the signature's typemark
is `nil`, so `id.type = chunk.signature.typename || @type` **always falls
back to the current `## ` H2 context** (`@type`, rrdparser.rb:636). In other
words: **only `gvar` is self-typing; `def`/`const`/`module_function def` are
indistinguishable to the parser and rely entirely on which H2 section they're
under.**

This was confirmed by 4 real corpus mismatches between the heading keyword
and the H2-derived kind (found via cross-tabulating the tool's keyword vs.
`kind` column):

| `GC.md:464` | `def INTERNAL_CONSTANTS` | `## Constants` (line 438) | `c` | matches real bitclust behavior — keyword typo in source, harmless |
| `RubyVM.md:55` | `def DEFAULT_PARAMS` | `## Constants` (line 38) | `c` | same |
| `functions_pp.md:1` | `def pp(*obj)` | `## Module Functions` (inherited via `functions.md` → `Kernel.md`) | `o` (=module_function) | correct — `Kernel#pp`/`Kernel.pp` really is a module function |

Not fixed here (out of scope) but worth a follow-up doctree issue. |

Kind mapping used (task only allows `s/i/c/v/o`, but bitclust's own internal

## 7. bitclust spec findings (for reference / traceability)

### (1) Where `{: since=}`/`{: until=}` is parsed and stored

- Attribute-line regex: `METHOD_ATTRIBUTE_LINE_RE = /\A\{:.*\}[ \t]*$/`
  (`bitclust/lib/bitclust/rrdparser.rb:274`).
- Parsed by `RRDParser::Context#method_attributes`
  (`rrdparser.rb:545-595`), called from `#define_method`
  (`rrdparser.rb:481-507`). Per-token key/value validated by
  `#parse_kv_method_attribute` (`rrdparser.rb:600-616`): key must be
  `since`/`until` (`KV_METHOD_ATTRIBUTES`, line 522), value must be quoted and
  match `KV_METHOD_ATTRIBUTE_VALUE_RE = /\A(?:\d+(?:\.\d+)*)?\z/` (line 523;
  the empty string is explicitly allowed by this regex).
- Attribute lines bind **only to the immediately preceding signature line**
  (`current_name`/`current_keys` reset on every new `--- `/`### ` line inside
  `#method_attributes`, rrdparser.rb:553-556); a blank line or body content
  ends the scan (line 577-578 `break`).
- Name resolution for binding: `#method_attribute_target_name`
  (rrdparser.rb:624-629) re-parses the signature line itself via
  `MethodSignature.parse`, stripping a leading `$` for special variables to
  match the DB's `names` convention.
- Storage: `MethodEntry` persistent properties `since_by_name`/`until_by_name`
  (`bitclust/lib/bitclust/methodentry.rb:114-120`), encoded as
  `"#{encodename_url(name)}=#{version}"` tokens (one array entry per name);
  decoded by `#decode_version_tokens` (lines 225-233) into `#since_map`/
  `#until_map` (lines 204-210). `#fill_since`/`#fill_until` (lines 214-221,
  and `#fill_version_token` at line 235) **only set a value if the name has
  none yet** — this is what lets explicit author values (written first, at
  parse time) permanently block the later automatic calculator (see below)
  from overwriting them, including the empty-string "explicitly unknown"
  case.

### (2) Badge decision logic (`bitclust/lib/bitclust/version_badges.rb`)

- `#heading_version_badges` (lines 34-42, used by `<dt>` method headings) and
  `#row_version_badges` (lines 49-55, used by the server's method-table rows)
  both consult `entry.since_map`/`entry.until_map`.
- For a heading with several aliases/overloads sharing the entry: if *every*
  name maps to the exact same version (`#uniform_map?`, lines 73-78 — `map.size
  == entry.names.size && map.values.uniq.size == 1`), the badge is rendered
  **once**, on the first signature's `<dt>` only; otherwise (mixed/partial —
  some names have a version and others don't, or they differ) each
  signature's own name is looked up individually (`#heading_badge_span`,
  lines 67-71).
- `#badge_span` (lines 80-84): returns no badge (`nil`) when the version is
  `nil` **or the empty string**. This is exactly how `{: since=""}` suppresses
  a badge — it writes the empty string into `since_by_name`, which both (a)
  displays nothing and (b) counts as "already set" for `#fill_since`, so the
  automatic calculator (below) can never overwrite it with a computed guess.
- Priority chain, highest to lowest: explicit non-empty `since="X"` (author) >
  explicit `since=""` (author-locked "no badge", still blocks autofill) >
  `MethodSinceCalculator`-computed value (only fills a name that has *no*
  value at all yet) > nothing (truly untouched → no badge).

### (3) `until=` semantics: "removed in Ruby X", not "existed through X"

- The badge text is generated from `UNTIL_CATALOG_KEY = 'removed in Ruby %s'`
  (`version_badges.rb:26`) — explicit confirmation that `until="X"` means the
  method **was removed starting at version X** (last version it existed in is
  the one immediately before X), not "existed up to and including X".
- Consistent with `MethodSinceCalculator#until_for`
  (`bitclust/lib/bitclust/method_since_calculator.rb:69-76`): computes the
  *next* ladder version after the last version in which the name was
  observed present (`@versions[idx + 1]`), i.e. the first version where it's
  gone.
- Also consistent with the `#@until VERSION` preprocessor directive itself:
  `preprocessor.rb:113-114` builds the condition `version < "VERSION"` — the
  guarded content applies only to versions *before* VERSION, i.e. the method
  (or behavior) is gone as of VERSION.
- The automatic calculator (`MethodSinceCalculator`, `method_since_calculator.rb`)
  scans a whole "version ladder" of per-version DBs to compute first/last
  appearance per `[class, typechar, name]` key (ignoring library), then
  writes results into each target DB via `#apply` (lines 83-115) using
  `fill_since`/`fill_until` — so it never overrides an already-authored value.
  A name present in the ladder's oldest version has `since_for` return `nil`
  (line 64-67, "floor" case — can't distinguish "always existed" from
  "existed before our data starts"), meaning no since-badge is shown; the doc
  corpus currently has **zero** explicit `until=` attributes (§2), so this
  entire mechanism is still purely automatic for `until` today.
