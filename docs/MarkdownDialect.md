# るりま Markdown と CommonMark・GFM の違い

るりまの原稿(`manual/` 配下の `.md`)は GitHub Flavored Markdown(GFM)をベースに、
リファレンスマニュアル用の拡張を加えた記法で書かれています。記法全体の仕様は
bitclust の [MARKUP_SPEC.md](https://github.com/rurema/bitclust/blob/master/doc/markdown-samples/MARKUP_SPEC.md)
にあります。このページは「CommonMark / GFM と**どこが違うか**」だけを 1 ページにまとめたものです。

処理系は bitclust(MDCompiler)であり、CommonMark の参照実装ではありません。
GitHub 上のプレビューと生成される HTML は、以下の点で異なることがあります。

## 独自拡張(CommonMark には無い意味を持つもの)

- **クロスリファレンス** `[m:Array#each]`・`[c:String]`・`[lib:json]`・`[ref:...]`・
  `[feature:N]`/`[bug:N]`・`[man:...]` など。
  CommonMark としては「定義の無いショートカット参照リンク」に当たる合法な形なので、
  GitHub などの汎用ビューアでは `[c:String]` というテキストのまま表示されます(壊れません)。
  将来、末尾に `[c:String]: URL` のような定義行を機械的に追記して汎用ツールでも
  リンクにできる余地を残した設計です。
  **ただし現在の bitclust は定義行(`[ラベル]: URL`)を特別扱いしないため、原稿に書かないでください**
  (定義行として消えず、そのまま表示に漏れます)。
- **見出し・定義リスト項目のアンカー** `### 見出し {#id}`。
  CommonMark ではただのテキストですが、るりまでは `id` 属性になり、
  `[ref:...]` やページ内リンクの参照先・検索インデックスの対象になります。
- **定義リスト** `- **用語**: 説明` は `<dl>`(dt/dd)に変換されます
  (CommonMark では通常の箇条書き)。`- **param**`・`- **return**`・`- **raise**`・
  `- **SEE**` などのキーワード形式も同じ仕組みの独自拡張です。
- **メソッドエントリ見出し** `### def name(args) -> Type` は
  メソッド定義として特別に解釈されます。
- **YAML front matter**(ファイル先頭の `---` ブロック)。CommonMark 本体には無い慣行です。
  ファイルの先頭に隙間なく置いたときだけ front matter として処理されます。
- **プリプロセッサ行** `#@since` / `#@until` / `#@end` / `#@include` / `#@#`(コメント)。
  bitclust がパース前に処理する独自機構です(汎用ビューアでは本文テキストに見えます)。
- **コードフェンスの属性** ```` ```ruby title="例" ```` や ```` ```ruby invalid ````。
  info string 自体は CommonMark として合法ですが、`title=`(キャプション表示)と
  `invalid`(構文チェックのスキップ)の意味付けは独自です。
- **生 HTML は書けません**。`<div>` などのタグ・`<`・`>`・`&` は常にエスケープされて
  そのまま表示されます(意図的な設計です)。
- **バックスラッシュエスケープは限定的**です。使えるのは `\[`・`` \` ``・行頭の `\#` のみで、
  `\*` や `\\` などはバックスラッシュごと表示に残ります。

## CommonMark / GFM の範囲内で使えるもの(注意付き)

- インラインリンク `[表示](URL)` — ただし宛先は `https://`(`http://`)と `#フラグメント` のみ。
  相対パスや `mailto:` は使えません(壊れた表示になります)
- オートリンク `<https://example.com>`
- GFM テーブル(アラインメント指定込み)
- コードスパン `` `code` ``・バッククォートフェンスのコードブロック
- **定義リスト(`- **用語**:`)の説明部分**は、空行を挟んだ複数段落やインデントした
  コードブロックを `<dd>` の中に正しく収められます

## 非対応(CommonMark / GFM にあるが使えないもの)

**多くはエラーにならず、「それらしく見えるが変換されないテキスト」として静かに劣化します。**
レビューで見逃しやすいので注意してください。

- **強調 `*em*` / `**strong**`**: 地の文では一切変換されず、アスタリスクがそのまま残ります。
  太字が効くのは定義リストの `**用語**`・キーワード(`**param**` 等)・行頭の `**N.**`
  (番号付き強調)という決まったパターンだけです
- チルダのコードフェンス `~~~` / 4 スペースインデントのコードブロック
  (コードブロックはバッククォートフェンスだけを使ってください)
- setext 見出し(`===` / `---` の下線)/ 水平線 `---` / 引用ブロック `>`
- ハードブレーク(行末 2 スペース・行末バックスラッシュ)
- 画像 `![alt](URL)`(`!` が残ってリンクとして誤解釈されます)
- 参照形式リンク `[表示][ラベル]` と定義行 `[ラベル]: URL`
- GFM 拡張の裸 URL 自動リンク・打ち消し線 `~~x~~`・タスクリスト `- [ ]`
- 順序付きリストの開始番号(`5.` から始めても 1 から採番されます)
- **通常の箇条書き(`- 項目`)で、空行を挟んだ継続内容**(段落やコードブロック)。
  リストの外に出て構造が壊れます。項目の中に複数段落やコードブロックが必要な場合は
  定義リスト形式(`- **用語**:`)を使ってください。インデント無しの継続行
  (CommonMark の lazy continuation)も非対応です

## 執筆時の指針

- 迷ったら**既存ファイルの書き方に合わせる**のが確実です
- 非対応記法の多くはビルドエラーにならないため、大きな変更では
  ローカルでのビルド([CONTRIBUTING.md](../CONTRIBUTING.md) 参照)や
  生成 HTML の確認をおすすめします
- コード例の構文エラーは CI が検出します(意図的な構文エラーの例には
  ```` ```text ```` または ```` ```ruby invalid ```` を使います)
